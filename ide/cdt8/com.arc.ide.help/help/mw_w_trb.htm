Troubleshooting

This section lists some common problems and their solutions. If the solutions listed do not solve your problem, go to http://solvnet.synopsys.com/ for application notes and technical support.

Problem: Can't page down to subsequent topics in help.
Cause: Eclipse Help does not offer paging between topics.
Solution: With most browsers and mouse devices, you can use the keyboard to page topics in the help:

  1. Give the Contents pane focus by clicking an entry.
  2. Use the up- or down-arrow key to highlight the previous or next topic the Contents pane (use the right- and left-arrow keys to expand and close categories).
  3. Use the Enter key to display the highlighted topic.

With most browsers and mouse devices, if a topic is longer than one screen, you can drag the scroll bar or use a mouse wheel to scroll up or down pages without removing focus from the Contents pane. Or you can repeatedly press the F6 key to shift focus between the Contents and help-topic panes, then use PgUp and PgDn to scroll.

Problem: Can't go to next topic in tutorial when I am performing a step in an IDE dialog.
Cause: The dialog windows are modal, that is, they cause the GUI to enter a dedicated mode.
Solution: On Windows, you can use the Task Manager to force minimization of the main IDE window.
Solution: Open two instances of the IDE.
Solution: Read ahead several steps in the tutorial before performing actions that open a dialog.

Problem: Build All and Build Project are greyed out on the Project menu.
Solution: Make sure you have a MetaWare toolkit installed (the location doesn't matter; the IDE looks for it by checking the PATH variable).

Problem: Building produces the following error: 

Error launching builder (make -k clean all )
    (Exec error:Launching failed)

Cause: The IDE cannot find gmake.
Solution: Make sure your MetaWare Development Toolkit is in ARC/MetaWare/arc, and make sure the toolkit's bin directory is in your path.

Problem: Build fails with an error such as the following:
'mcc' is not recognized as an internal or external command.
Cause: Toolset for the specified target processor is no longer available (removed).

Problem: Build fails with an error such as the following: 

gmake -k clean all
gmake: *** No rule to make target `clean'.
gmake: *** No rule to make target `all'.

Cause: No makefile exists in a makefile project.
Solution: Add a makefile to your project.

Problem: Toolset sometimes rejects hyphens in project name.
Solution: Rename the project without hyphens (right-click the project and select Rename). Spaces are acceptable.

Problem: Debugger windows not synchronized, source changes not apparent in program output.
Cause: If you rebuild while debugging, the debugger continues debugging the old executable until you restart the executable.
Solution: Click Restart .

Problem: Breakpoint disappears when I try to select it in Debugger -> Breakpoints.
Problem: Breakpoint behavior no longer consistent between Debugger -> Breakpoints and Window > Show View > Breakpoints.
Cause: If you rebuild while debugging, then restart, the debugger engine detects that the file has changed and attempts to re-apply breakpoints at the same source location. If a source file has lines added or deleted, the breakpoints might no longer be in expected places.
Solution: Delete and re-set the breakpoints.

Cause: The IDE is invoking your toolset (mcc) to capture output and deduce the predefined macros and include files. Possibly mcc is returning a non-zero return code.
Solution: You can safely ignore this error.

Problem: The IDE errors out when I select Run > Debug and expand C/C++ Local Application.
Solution: Select your ELF file, not just your project, under Binaries.

Problem: When performing some operations, I get a Windows security alert:

Cause: Windows often issues this alert when running new Java programs.
Solution: Click Unblock when this alert is generated by the IDE.

Problem: Debugger crashes when Call Stack window displays many calls.
Cause: Calculation of the call stack might be looping.
Solution: Manually specify the maximum number of stack frames to display using command-line option -max_stack_depth=N.

Problem: Building my standard-make project fails with errors about some commands not being found.
Cause: Some copies of make or gmake use backlashes as escape characters, and some use backslashes as directory separators. If a directory separator is interpreted as an escape, the Build command sees only a long string of characters where a path was intended.
Solution: Make sure you have the latest IDE patch installed, and use the gmake supplied with it.
